# PAIS CLI v0.4.0 — Unified `pais`, generic ingest, declarative indexes

## Context

After landing v0.3.x the user hit three remaining sharp edges running real PAIS:

1. **Two binaries (`pais` + `pais-dev`) hide the most-used commands.** `pais --help` doesn't show the feed pipeline at all, so the user couldn't find how to put data into a KB.
2. **Names are suite-specific.** `pais-dev ingest-suites` reads as "only for test suites." User wants a generic feed that picks the right splitter per index.
3. **Every command needs raw UUIDs** and the splitter behavior is hard-coded. User wants to declare KBs + indexes + splitter config in TOML once, then operate on short aliases. Plus: `pais kb list` should show docs counts and human dates; `pais kb show` should exist.

This plan unifies the CLI, makes ingest generic with a per-index splitter declared in config, and adds an alias system + `pais kb ensure` so the TOML becomes the source of truth.

## End-state UX

```toml
# ~/.pais/config.toml
[profiles.lab]
mode = "http"
base_url = "https://10.160.11.45/api/v1"
auth = "none"
verify_ssl = false

[profiles.lab.knowledge_bases.test_suites]
name = "mops-permanent-test-suites"
data_origin_type = "LOCAL_FILES"

  [[profiles.lab.knowledge_bases.test_suites.indexes]]
  alias = "main"
  name = "ts-idx"
  embeddings_model_endpoint = "BAAI/bge-small-en-v1.5"
  chunk_size = 512
  chunk_overlap = 64

    [profiles.lab.knowledge_bases.test_suites.indexes.splitter]
    kind = "test_suite_md"   # H1/H2/H3 atomic sections + breadcrumb header
    budget_tokens = 400

  [[profiles.lab.knowledge_bases.test_suites.indexes]]
  alias = "raw"
  name = "ts-raw"
  embeddings_model_endpoint = "BAAI/bge-small-en-v1.5"
  chunk_size = 512
  chunk_overlap = 64

    [profiles.lab.knowledge_bases.test_suites.indexes.splitter]
    kind = "passthrough"     # upload files as-is; PAIS handles splitting
```

```bash
# day 0 — make PAIS match the file
pais --profile lab kb ensure          # creates anything missing, idempotent

# day-to-day
pais ingest test_suites:main ./suites/         # splitter pulled from config
pais ingest test_suites:raw  ./pdfs/           # different splitter, same KB
pais ingest test_suites:main ./changed/ --replace
pais kb list --with-counts                     # short-name + doc counts
pais kb show test_suites                       # full breakdown per index
```

`pais-dev` is gone. Everything lives under `pais`.

## Decisions

- **Single binary**: `pais-dev` deleted. Its three commands (`split-suite`, `ingest-suite`, `ingest-suites`) merge into one generic `pais ingest`. Console-script `pais-dev` removed; first invocation of the old name prints a one-line redirect for users who still have it on PATH from an older install (a tiny shim entry, removed in v0.5).
- **Generic ingest, splitter from config**: `pais ingest <alias> <path>` looks up the index, reads its `[splitter]` block, runs that splitter, uploads the result. CLI `--splitter <kind>` overrides config (testing/dev only).
- **Splitter registry**: a `Splitter` protocol + four built-ins shipped in v0.4 — `test_suite_md` (today's), `markdown_headings` (generic H1/H2/H3, no suite assumptions), `passthrough` (upload as-is), `text_chunks` (simple sentence/char chunking for plain `.txt` / `.log`). Adding a 5th is one file.
- **Alias system**: TOML keys (`test_suites`, `main`) are local short-names. `pais` resolves them to UUIDs by name on first use, caches in `~/.pais/aliases.json`, invalidates cache entry on a 404. UUIDs continue to work everywhere — `pais kb show <uuid>` and `pais kb show test_suites` both work.
- **`pais kb ensure`**: idempotent; reads the active profile's TOML, creates KBs/indexes that don't exist on the server. Updates that PAIS doesn't allow (chunk_size, embeddings_model_endpoint) emit a warning instead of mutating. `--dry-run` prints intended actions; `--prune --yes` deletes KBs/indexes on the server that aren't in the TOML (off by default).
- **`kb list --with-counts`** is **opt-in** (300 KBs × 2 round trips is a real cost). Default columns add `description` + `last_updated_at` (no extra calls).
- **`pais kb show <kb>`** is the new one-shot detail view. Always fetches indexes (1 extra call per KB).
- **Human dates by default**, `--epoch` opt-out for scripts.
- **Per-index slug for `--replace`**: each splitter exposes `group_key(input_path) -> str`. `test_suite_md` returns the H1 slug (today's behavior); `passthrough` returns the filename stem; `markdown_headings` returns H1 or stem; `text_chunks` returns stem. `--replace` matches `origin_name` prefixes accordingly.
- **No data routing by file extension** in v0.4. One splitter per ingest call. If you need to split `.md` and `.pdf` differently, run `pais ingest` twice with different aliases.

## TOML schema (locked)

```
[profiles.<name>.knowledge_bases.<kb_alias>]
name              = "..."     # required, real PAIS KB name
description       = "..."
data_origin_type  = "LOCAL_FILES" | "DATA_SOURCE"
index_refresh_policy = { policy_type = "MANUAL" }   # optional

[[profiles.<name>.knowledge_bases.<kb_alias>.indexes]]
alias                     = "main"     # required, local short-name
name                      = "ts-idx"   # required, real PAIS index name
embeddings_model_endpoint = "..."
text_splitting            = "SENTENCE"
chunk_size                = 512
chunk_overlap             = 64

  [profiles.<name>.knowledge_bases.<kb_alias>.indexes.splitter]
  kind          = "test_suite_md" | "markdown_headings" | "passthrough" | "text_chunks"
  budget_tokens = 400              # for splitters that enforce a budget
  # kind-specific options live here:
  # for `text_chunks`: chunk_chars = 1500, overlap_chars = 100
  # for `markdown_headings`: heading_level = 2, breadcrumb = true
```

Validation happens at TOML-load time via pydantic models (one per splitter kind). Errors print the TOML path that's wrong (`profiles.lab.knowledge_bases.test_suites.indexes[0].splitter.budget_tokens`).

## New CLI surface (final)

```
pais ingest <alias|uuid_kb>:<alias|uuid_index> <path> [options]
    --splitter <kind>      override the config splitter
    --replace              delete existing docs whose origin_name matches the
                            current splitter's group_key, then upload
    --workers N            default 4
    --report PATH          JSON report (default ./ingest-report.json)
    --dry-run              split locally, print summary + token distribution,
                            upload nothing
    --output table|json|yaml

pais kb list [--with-counts] [--epoch]
pais kb show <alias|uuid> [--epoch]
pais kb ensure [--dry-run] [--prune --yes]
pais splitters list                  # all registered splitter kinds + schema
pais splitters show <kind>           # one kind's docs + option schema
pais alias list                      # cached UUID resolutions, by profile
pais alias clear [<alias>]           # invalidate one or all cache entries

# Existing commands accept aliases anywhere they took UUIDs:
pais index list test_suites
pais index search test_suites:main "..."
pais kb purge test_suites --yes
pais index cancel test_suites:main --yes
```

## Splitter abstraction

```python
# src/pais/ingest/splitters/_base.py
class Splitter(Protocol):
    kind: ClassVar[str]                       # registry key
    options_model: ClassVar[type[BaseModel]]  # pydantic schema for TOML opts

    def __init__(self, options: BaseModel) -> None: ...

    def split(self, path: Path) -> Iterator[SplitDoc]:
        """Yield one upload-ready SplitDoc per output chunk-file."""

    def group_key(self, path: Path) -> str:
        """Return the slug used by --replace to match origin_name prefixes."""

@dataclass
class SplitDoc:
    origin_name: str          # filename to register with PAIS
    body: bytes               # exact bytes to POST
    media_type: str = "text/markdown"
    metadata: dict[str, Any] = field(default_factory=dict)  # for the report only
```

A `SPLITTER_REGISTRY: dict[str, type[Splitter]]` with the four built-ins. The registry is open — third parties can register more, but the v0.4 surface is intentionally small.

The four built-ins:

| kind | input | output behavior |
|---|---|---|
| `test_suite_md` | one .md per suite, structured with H1/H2/H3 | today's per-section atomic split + breadcrumb. `group_key` = H1 slug. |
| `markdown_headings` | any .md | split at configurable heading level; breadcrumb header optional. `group_key` = H1 or stem. |
| `passthrough` | any file | one chunk per file, body = file contents, no transformation. `group_key` = stem. |
| `text_chunks` | plain text / log | character-based chunking with overlap. `group_key` = stem. |

`test_suite_md` is the existing implementation moved + renamed (no behavior change).

## Alias resolver

`src/pais/cli/_alias.py`:

```python
def resolve_kb(client, profile, kb_ref) -> str:
    """kb_ref is either a UUID or a TOML alias. Returns UUID."""

def resolve_index(client, profile, kb_ref, idx_ref) -> tuple[str, str]:
    """Returns (kb_uuid, index_uuid). Both refs may be aliases or UUIDs."""
```

Cache file: `~/.pais/aliases.json` shaped as:

```json
{
  "lab": {
    "kbs": {"test_suites": {"uuid": "...", "name": "..."}},
    "indexes": {
      "test_suites:main": {"uuid": "...", "kb_uuid": "...", "name": "..."}
    }
  }
}
```

Invalidation: any 404 on a previously-cached UUID → drop that entry, re-resolve, retry the original call once. `pais alias clear` wipes manually.

UUID detection is pragmatic: anything matching `[0-9a-f]{8}-[0-9a-f]{4}-...` (UUID4 shape) is treated as a UUID; anything else is an alias. Alias keys must match `[A-Za-z][A-Za-z0-9_-]*` to keep the regex unambiguous.

## `pais kb ensure` flow

For the active profile:

1. List server-side KBs once. Build `{name → uuid}` map.
2. For each TOML KB: if `name` exists → record; else → POST create.
3. For each KB: list its indexes, build `{name → uuid}` map.
4. For each TOML index under that KB: if `name` exists → check declared `embeddings_model_endpoint`/`chunk_size`/`chunk_overlap` match what the server has; warn on mismatch (PAIS doesn't expose update for these); else → POST create.
5. Refresh the alias cache for the profile.
6. Print a per-row summary (✓ existing, + created, ⚠ mismatch).

`--dry-run`: steps 1–4 only, no POSTs, prints what would happen.
`--prune --yes`: also removes server-side KBs/indexes whose `name` is not in the TOML. Hard-gated, separate confirmation per item.

## Safety review (risks + mitigations)

| Risk | Mitigation |
|---|---|
| Alias accidentally looks like a UUID | UUID detected by strict 8-4-4-4-12 hex regex first. Aliases must match `[A-Za-z][A-Za-z0-9_-]*` AND not match the UUID regex. Loader rejects ambiguous aliases. |
| Stale alias cache after server-side delete/recreate | Any 404 on a cached UUID drops that entry, re-resolves, retries the original call once. Manual escape: `pais alias clear`. |
| `pais kb ensure --prune` deletes the wrong thing | Per-item confirmation (each KB / index prompted separately), even with `--yes`. `--prune` cannot run without `--yes`. `--dry-run` is the discoverable preview. |
| Worker pool races on alias resolver | Aliases are resolved serially **before** the worker pool starts; workers receive UUIDs only. No shared mutable state in workers. |
| `--replace` matches the wrong docs across mixed splitters | `--replace` deletes by `splitter.group_key(input_path)` per file, not by stripping the input dir. Different splitters yield different keys → no cross-contamination. |
| `pais-dev` callers in user scripts break silently on upgrade | v0.4 ships a `pais-dev` shim that prints a one-line redirect and exits 1 (loud failure, not silent no-op). Removed in v0.5. CHANGELOG calls this out under **Breaking changes**. |
| Splitter `kind` typo in TOML silently uses wrong splitter | Loader validates `kind` against the registry at load time and points at the exact TOML path (`profiles.lab.knowledge_bases.X.indexes[N].splitter.kind`). |
| Splitter options have wrong type in TOML | Each splitter declares an `options_model` (pydantic). Loader validates options against it at load time, not at ingest time. |
| `pais kb ensure` on a fresh PAIS server creates duplicates if a KB with same name exists | Lookup is by `name` first, by UUID never. Same-name match → reuse, no create. Mock backend extended to support lookup-by-name on indexes too. |
| Embeddings-model / chunk-size mismatch between TOML and live index | PAIS doesn't expose update for these. `ensure` warns (does NOT mutate, does NOT recreate). User has to delete + recreate manually. CLI prints the exact diff. |
| Cache file `~/.pais/aliases.json` corrupted | Corruption → start with empty cache, log a warning. Never blocks a command. |
| Concurrent `pais kb ensure` runs race | Single-process tool; documented as not safe for concurrent invocation. No file lock added in v0.4 (out of scope). |
| Generic `passthrough` splitter uploads sensitive files by accident | Same risk as `pais index upload` today. README "content hygiene" note already documents this; reinforced in the new ingest section. |

## Out of scope (Phase 5)

- Per-file routing by glob/extension within one ingest call.
- Splitters declared in user code (third-party plug-in registration via entry-points). The registry is open in code; the CLI doesn't yet load external entry-points.
- `pais kb ensure --update` that mutates server config to match TOML when PAIS exposes an update verb.
- Server-side aliases (PAIS doesn't expose them).
